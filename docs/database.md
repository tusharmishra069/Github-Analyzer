# Database

## Overview

The application uses **Neon Postgres** — a serverless PostgreSQL provider — as its sole persistent store. The database is accessed synchronously via **SQLAlchemy 2.x** with a connection pool tuned for Neon's PgBouncer-based pooler.

---

## Connection Configuration

Defined in `backend/database.py`:

```python
engine = create_engine(
    DATABASE_URL,          # postgresql://... (Neon pooled endpoint)
    pool_size=5,           # base connections kept open
    max_overflow=10,       # burst headroom on top of pool_size
    pool_timeout=30,       # seconds to wait before raising PoolTimeout
    pool_recycle=1800,     # recycle connections after 30 min (avoids stale)
    pool_pre_ping=True,    # validate connection before use
)
```

The `DATABASE_URL` must use the **pooled** Neon endpoint (`...neon.tech/neondb?sslmode=require`). The app rewrites any `postgres://` prefix to `postgresql://` automatically for SQLAlchemy compatibility.

Sessions are created per-request via:

```python
SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)
```

## Schema

### `jobs` table

This is the only table in the database. It tracks the lifecycle of every repository analysis request.

| Column | Type | Constraints | Description |
|---|---|---|---|
| `id` | `VARCHAR(36)` | PK, not null | UUID v4 — generated by the backend at request time |
| `repository_url` | `VARCHAR(255)` | indexed, not null | Full GitHub repo URL as submitted by the user |
| `status` | `VARCHAR(50)` | not null | Current job state (see state machine below) |
| `progress` | `INTEGER` | default 0 | Completion percentage 0–100 |
| `message` | `VARCHAR(255)` | nullable | Human-readable status message shown in the UI |
| `result` | `JSON` | nullable | Full LLM analysis output; populated on completion |
| `created_at` | `DATETIME` | server_default=now | Timestamp when the job was created |
| `updated_at` | `DATETIME` | onupdate=now | Timestamp of last status change |

**SQLAlchemy model** (`backend/models.py`):

```python
class Job(Base):
    __tablename__ = "jobs"

    id             = Column(String(36), primary_key=True)
    repository_url = Column(String(255), nullable=False, index=True)
    status         = Column(String(50), nullable=False)
    progress       = Column(Integer, default=0)
    message        = Column(String(255))
    result         = Column(JSON, nullable=True)
    created_at     = Column(DateTime, server_default=func.now())
    updated_at     = Column(DateTime, onupdate=func.now())
```

## Job State Machine

```
                    POST /api/analyze
                           │
                           ▼
                       ┌─────────┐
                       │ PENDING │   Row created, background task queued
                       └────┬────┘
                            │  Worker picks up task
                            ▼
                    ┌────────────────┐
                    │  PROCESSING    │   progress: 0 → 100
                    │                │   message updated at each milestone
                    └───┬────────┬───┘
               success  │        │  exception
                        ▼        ▼
                  ┌──────────┐  ┌────────┐
                  │COMPLETED │  │ FAILED │
                  │result≠null│ │result=null│
                  └──────────┘  └────────┘
```

### Progress Milestones

| Progress | Message |
|---|---|
| 10 | Cloning repository... |
| 25 | Parsing source files... |
| 40 | Building knowledge base... |
| 50 | Retrieving relevant context... |
| 70 | Generating AI analysis... |
| 88 | Structuring results... |
| 100 | Analysis complete |

## `result` JSON Schema

When a job reaches `COMPLETED`, the `result` column contains a JSON object with the following structure:

```jsonc
{
  "health_score": 82,                  // integer 0–100
  "summary": "...",                    // 2–3 sentence overview
```

## Data Lifecycle

- **Retention**: Jobs are never automatically deleted. All rows persist indefinitely.
- **Re-analysis**: Submitting the same URL creates a new Job with a fresh UUID. No deduplication is performed.
- **Profile Review / Roast**: These features are stateless — no data is written to the database. Only the `jobs` table is used (for repository analysis only).

---

## Running Migrations

The schema is created at application startup via SQLAlchemy's `create_all`:

```python
# database.py
Base.metadata.create_all(bind=engine)
```

This is idempotent — safe to run multiple times. For production schema changes, use Alembic or Neon's branching feature to test migrations before applying to the main branch.
